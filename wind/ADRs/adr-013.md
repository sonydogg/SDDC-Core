# ADR-013: Secure Secret Management for Perforce

**Status:** Accepted (2026-03-16)

**Context:**
The SDDC Core requires secure authentication for the Perforce Librarian (sddc-p4d). Storing passwords in plain text within Docker environment variables or shell history is a security risk.
* **Constraint A:** Use Docker's "File-based Secret" pattern for the P4 admin password.
* **Constraint B:** Maintain the P4USER as 'admin' for consistency across the Intel Mini cluster.
* **Constraint C:** Ensure the `secrets/` directory is excluded from Git to prevent leakage to GitHub.

**Decision:**
We will utilize the `P4PASSWD_FILE` environment variable within the `sddc-p4d` container:
1. **The Secret:** The admin password is stored in `./secrets/p4_admin_pass` on the host.
2. **The Mount:** This file is mounted as a read-only volume to `/run/secrets/p4_admin_pass` inside the container.
3. **Authentication:** Client-side authentication (VS Code/CLI) will rely on `p4 login`, which generates a local ticket, rather than storing the password in `.p4config`.

**Consequences:**
* **Positive:** High security; the password never appears in `docker inspect` or process lists.
* **Positive:** Centralized password management in the `./secrets` folder.
* **Negative:** Requires a manual `p4 login` step whenever the login ticket expires.
* **Negative:** Losing the local `./secrets` folder would lock the administrator out of the depot.

---

**Technical Implementation:**
Environment variables in `docker-compose.yaml`:
- `P4USER=admin`
- `P4PASSWD_FILE=/run/secrets/p4_admin_pass`
