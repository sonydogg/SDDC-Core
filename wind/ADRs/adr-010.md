# ADR-010: Perforce as the Image Factory Source of Truth

**Status:** Proposed (2026-03-16)

**Context:**
The transition to "Declarative Image Management" (ADR-009) introduces a requirement for managing large binary assets (ISOs, WIMs, VHDs) and complex build scripts. GitHub is ill-suited for these heavy payloads. As an architect, we require a system that provides versioning, auditability, and high-performance local access to these assets.
* **Constraint A:** Store and version Windows ISOs and large driver packages.
* **Constraint B:** Maintain a high-speed link between the code and the "Water" stone storage.
* **Constraint C:** Ensure build reproducibility by locking specific binary versions to build scripts.

**Decision:**
We will utilize **Perforce (sddc-p4d)** as the primary repository for the Image Factory:
1. **The Depot:** A new depot `//depot/images/...` will be created on the `water` stone to house all Packer templates and binary sources.
2. **Versioning Strategy:** Every "Release" of an image (e.g., Windows 11 - Frederick Freeze Edition) will be captured as a labeled Perforce changelist containing both the code and the specific binary versions used.
3. **Local Workspace:** The build runner on the Intel Mini will use a dedicated P4 workspace mapped to `/mnt/stones/wind/build_workspace` for high-speed IO during the bake process.

**Consequences:**
* **Positive:** Complete local control over large assets without external bandwidth or storage limits.
* **Positive:** "The Librarian" provides a clear history of which ISO version was used for which machine build.
* **Negative:** Requires active management of Perforce metadata and depots on the `earth` and `water` stones.
* **Negative:** Disconnects the "Binary Build Code" from the "Public ADRs" on GitHub, requiring the hybrid bridge discussed in ADR-008.

---

**Technical Implementation:**
Perforce streams or branching will be used to manage "Base OS" layers versus "Specialized Application" layers (e.g., Photography vs. SysAdmin builds).
