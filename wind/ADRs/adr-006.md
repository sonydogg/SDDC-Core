# ADR-006: Containerized Volume Mapping Strategy for SDDC Core

**Status:** Accepted (2026-03-16)

**Context:**
The SDDC Core utilizes Docker Compose to manage the "Fifth Element" services (Perforce, n8n, Postgres). To maintain the "Stones" architectural integrity, container volumes must be mapped to specific logical directories that match the data's durability and performance requirements.
* **Constraint A:** Database files must reside on "Earth" (Persistent/Safe).
* **Constraint B:** High-churn data and configuration files must reside on "Wind" (Active/Ephemeral).
* **Constraint C:** Large data repositories (Perforce Depots) must reside on "Water" (Mass Storage).
* **Constraint D:** Avoid the use of the `&` character in naming and documentation to prevent n8n Markdown-to-HTML parsing errors.

**Decision:**
We have mapped the Docker service volumes to the `/mnt/stones/` hierarchy as follows:

1. **Service: sddc-p4d (Perforce)**
   * Metadata: `/mnt/stones/earth/p4data` mapped to `/data`
   * Depots: `/mnt/stones/water/p4depots` mapped to `/data/depots`

2. **Service: sddc-n8n (Automation Brain)**
   * Active Config: `/mnt/stones/wind/n8n_data` mapped to `/home/node/.n8n`

3. **Service: sddc-n8n-db (The Memory)**
   * Postgres Data: `/mnt/stones/earth/n8n_db_data` mapped to `/var/lib/postgresql/data`

4. **Service: sddc-proxy (Nginx Proxy Manager)**
   * Runtime Data: `/mnt/stones/wind/proxy_data` mapped to `/data`
   * Certificates: `/mnt/stones/wind/proxy_letsencrypt` mapped to `/etc/letsencrypt`

**Consequences:**
* **Positive:** Clear separation of concerns; "Earth" can be backed up differently than "Wind".
* **Positive:** Scaling storage for a specific stone (e.g., adding an external drive for "Water") does not require changing container internal paths.
* **Negative:** Service startup depends on the availability of the `/mnt/stones/` mount points.
* **Negative:** Cross-stone dependencies (e.g., n8n-db on Earth and n8n-config on Wind) mean a failure in one stone's directory structure may impact multiple services.

---

**Technical Implementation:**
All service mappings are managed via the `sddc-core` docker-compose.yaml file. Secret management is handled via file-based secrets located in `./secrets/` and mapped as read-only volumes to prevent sensitive data leakage.
