# ADR-009: Transition to Declarative Image Management

**Status:** Proposed (2026-03-16)

**Context:**
The SDDC Core currently relies on manual or commodity imaging tools (SCCM/WDS/Ninite equivalents). These are "Black Box" solutions that are difficult to version control, audit, or reproduce. Game development environments and cloud providers (AWS) use dynamic, code-based builds that allow for complex, repeatable environments.
* **Constraint A:** Move away from monolithic "Golden Images."
* **Constraint B:** Version control the build process in GitHub/Perforce.
* **Constraint C:** Decouple the Base OS from the Application Stack.

**Decision:**
We will adopt a **Code-First Image Factory** model using HashiCorp Packer and Ansible:
1. **Base Layer:** Packer will build the "Raw Stone" OS from a clean ISO, ensuring a consistent patched baseline.
2. **Provisioning Layer:** Ansible or advanced PowerShell scripts will handle application installation (via Winget/Chocolatey) and OS hardening.
3. **Artifact Storage:** Finished templates will be stored on the `water` stone to prevent re-downloading large ISOs.

**Consequences:**
* **Positive:** Every change to the "Image" is a Git commit.
* **Positive:** Removing the need for SCCM/WDS infrastructure reduces resource overhead on the Intel Mini.
* **Negative:** Requires a steep learning curve for Packer/Ansible syntax.
* **Negative:** Initial "Build" time is longer than a standard clone, as the OS is built from scratch.

---

**Technical Implementation:**
Initial builds will target VMware environments on the Intel Mini, utilizing the existing `water` stone for storage of ISOs and build artifacts.
