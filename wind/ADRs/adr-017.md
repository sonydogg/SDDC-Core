# ADR-017: Ephemeral Tooling vs. Declarative Injection

**Status:** Accepted (2026-03-16)

**Context:**
The `sddc-p4d` container is an immutable, minimalist environment. Installing persistent tools like `vi` or `nano` inside the container violates the principle of "Infrastructure as Code," as these tools disappear during container recreation or image updates. 
* **Constraint A:** Maintain the "Librarian" without persistent modifications to the container OS.
* **Constraint B:** Enable administrative tasks (Protections, User edits) in an editor-less environment.
* **Constraint C:** Avoid "Configuration Drift" caused by manual internal container changes.

**Decision:**
We will prioritize **Declarative Injection** over ephemeral tool installation:
1. **Piping Pattern:** All administrative updates will be handled via `echo "..." | p4 [command] --replace` or `-i`.
2. **External Editing:** For complex files, the architect will author the text in VS Code (on the host) and pipe the file content into the container: `cat file.txt | docker exec -i sddc-p4d p4 [command] -i`.
3. **Emergency Tooling:** If an interactive editor is strictly required, it will be installed as a temporary layer using `apt-get` or `apk` with the understanding that it is non-persistent.

**Consequences:**
* **Positive:** Keeps the SDDC Core clean and predictable.
* **Positive:** Forces all configuration changes to be scripted, making them easier to add to ADRs.
* **Negative:** Slightly higher "Syntax Tax" for the architect when running commands.

---

**Technical Implementation:**
To update the Protections table via a file on the host:
`cat protections.txt | docker exec -i sddc-p4d p4 protect --replace`
