# ADR-024: Honcho Self-Hosted LLM Stack — Hardening, Failover, and Operational Lessons

**Date:** 2026-06-11  
**Status:** Accepted  
**Deciders:** Trevor Long  
**Related:** ADR-008 (Ernie Agent), SDDC-56, SDDC-57, SDDC-58, SDDC-59, SDDC-60

---

## Context

On 2026-06-11, the gaming desktop (momoney, 192.168.5.139) went offline unexpectedly while Honcho's self-hosted LLM stack was in active use. The outage exposed four independent provisioning and configuration gaps that had never been exercised:

1. **Embedding 401s** — Honcho had been configured with a dummy OpenAI API key since initial deployment. Every embedding call was silently failing with a 401. Zero embeddings had ever been stored.
2. **Groq fallback not firing** — The fallback config blocks in `.env` were present but missing the `API_KEY_ENV` pointer that tells the resolver which environment variable holds the key. `_resolve_fallback_config()` in `config.py` requires this field explicitly; without it, fallback is silently skipped.
3. **LLM timeout hardcoded at 600s** — `registry.py` had `timeout=600.0` baked in. Under VRAM contention (e.g., Minecraft running on momoney), Ollama can stall for 10–60s on model reload. A 600s hang means silent failure for nearly 10 minutes before any fallback logic is reached.
4. **Dialectic levels `medium`/`high`/`max` unconfigured** — `DialecticSettings` in `config.py` has a hardcoded default that falls through to `gpt-5.4-mini` when a level is not configured. Only `minimal` and `low` were wired. All deeper reasoning calls were hitting a non-existent OpenAI model and receiving 401s.

All four issues were silent. No crash, no alert, no fallback — just degraded or absent Honcho capability.

---

## Decision

Apply targeted fixes to `/opt/honcho/.env` and `/opt/honcho/src/llm/registry.py` to resolve all four gaps, then rebuild the stack. No architectural changes — Ollama on momoney remains primary; Groq (`llama-3.3-70b-versatile`) remains emergency cloud fallback.

---

## Changes Made

### 1. Embeddings — switched to local `nomic-embed-text` via Ollama

**Root cause:** Dummy `LLM_OPENAI_API_KEY` caused all embedding calls to fail with 401.

**Fix:** Added `EMBEDDING_MODEL_CONFIG__*` vars to `.env` pointing at `nomic-embed-text` on Ollama. `nomic-embed-text` produces 768-dim vectors; the pgvector schema had been provisioned at 1536-dim (OpenAI default). Dropped HNSW indexes, ALTERed both `message_embeddings.embedding` and `documents.embedding` columns from `vector(1536)` to `vector(768)`, and recreated HNSW indexes. Safe to do because zero rows had ever been stored.

```dotenv
EMBEDDING_MODEL_CONFIG__TRANSPORT=openai
EMBEDDING_MODEL_CONFIG__MODEL=nomic-embed-text
EMBEDDING_MODEL_CONFIG__OVERRIDES__BASE_URL=http://192.168.5.139:11434/v1
EMBEDDING_VECTOR_DIMENSIONS=768
```

**Validation:** Confirmed real 768-dim float vector in API response; zero 401s.

### 2. Groq fallback — added missing `API_KEY_ENV` pointer to all four modules

**Root cause:** `_resolve_fallback_config()` in `src/config.py` requires an explicit `api_key_env` field on the fallback block. All four module fallback stanzas (api, deriver, and associated sub-configs) were missing this pointer.

**Fix:** Added `FALLBACK__OVERRIDES__API_KEY_ENV=LLM_GROQ_API_KEY` to every fallback block:

```dotenv
# Example pattern applied to all four blocks
<MODULE>__MODEL_CONFIG__FALLBACK__OVERRIDES__API_KEY_ENV=LLM_GROQ_API_KEY
```

**Validation:** `select_model_config_for_attempt(mc, 3)` returns correct Groq config in running container.

### 3. LLM timeout — patched `registry.py` from 600s to 45s

**Root cause:** Hardcoded `timeout=600.0` in all client factory functions in `src/llm/registry.py`.

**Fix:** Added a module-level constant driven by environment variable, applied to all client factories:

```python
_LLM_REQUEST_TIMEOUT = float(os.environ.get("LLM_REQUEST_TIMEOUT", "45"))
```

Added `LLM_REQUEST_TIMEOUT=45` to `.env`.

**Failover math:** Healthy Ollama responds in 1–10s. Model reload under VRAM contention: ~10–30s. With 45s timeout × 2 attempts + ~24s backoff, Groq fires at approximately 114s — acceptable for background agent work.

**Validation:** Confirmed `_LLM_REQUEST_TIMEOUT = 45.0` in running container.

### 4. Dialectic levels — wired `medium`, `high`, `max` to Ollama with Groq fallback

**Root cause:** `DialecticSettings` falls through to hardcoded `gpt-5.4-mini` default when a level is not configured. Only `minimal` and `low` had been set; `medium`, `high`, and `max` were unconfigured. An `xhigh` entry was also incorrectly added — `xhigh` exists in the code's `ReasoningLevel` Literal but is **not** a valid key in `DialecticSettings` and causes a `ValidationError` on startup.

**Fix:** Added `DIALECTIC_LEVELS__medium/high/max__MODEL_CONFIG__*` blocks to `.env`. Removed the invalid `xhigh` block.

Valid levels for `DialecticSettings`: `minimal`, `low`, `medium`, `high`, `max`.

```dotenv
DIALECTIC_LEVELS__medium__MODEL_CONFIG__TRANSPORT=openai
DIALECTIC_LEVELS__medium__MODEL_CONFIG__MODEL=qwen2.5:14b-instruct-q4_K_M
DIALECTIC_LEVELS__medium__MODEL_CONFIG__OVERRIDES__BASE_URL=http://192.168.5.139:11434/v1
DIALECTIC_LEVELS__medium__MODEL_CONFIG__FALLBACK__TRANSPORT=openai
DIALECTIC_LEVELS__medium__MODEL_CONFIG__FALLBACK__MODEL=llama-3.3-70b-versatile
DIALECTIC_LEVELS__medium__MODEL_CONFIG__FALLBACK__OVERRIDES__BASE_URL=https://api.groq.com/openai/v1
DIALECTIC_LEVELS__medium__MODEL_CONFIG__FALLBACK__OVERRIDES__API_KEY_ENV=LLM_GROQ_API_KEY
# Repeat pattern for high and max
```

**Validation:** Stack starts clean, zero ValidationErrors in API logs.

## Gotchas and Lessons Learned

### `docker compose restart` does NOT reload `.env`
`restart` reuses the existing container environment. Any `.env` change requires `docker compose down && docker compose up -d`. This will always catch you once.

### `xhigh` is not a valid `DialecticSettings` key
`xhigh` appears in the `ReasoningLevel` Literal type in `config.py` but is explicitly excluded from `DialecticSettings` validation. Adding it to `.env` causes a `ValidationError` on startup and the API container will loop-crash. Valid keys: `minimal`, `low`, `medium`, `high`, `max` only.

### Groq fallback silence
Groq fallback looks configured when it isn't. All four module blocks need the `API_KEY_ENV` pointer. The resolver does not warn on missing key — it simply skips fallback and lets the primary fail silently.

### Fallback is emergency-only — not a primary LLM
Groq free tier: 30 RPM / 1K RPD / 100K TPD. Developer tier was at capacity at time of writing. Groq is a safety net for transient Ollama failures, not a sustained workload path.

### VRAM contention is real
When Minecraft (or any GPU-heavy workload) is running on momoney, Ollama evicts models from VRAM and must reload them on next call. Reload time observed: 9–30s. The 45s timeout provides adequate headroom. If gaming workloads become regular, consider a Hermes-side `OLLAMA_KEEP_ALIVE` tuning or a session-aware routing rule.

### Embedding schema must match model dimensions
`nomic-embed-text` produces 768-dim vectors. OpenAI `text-embedding-ada-002` produces 1536-dim. HNSW indexes are dimension-bound — a mismatch causes silent storage failure or query errors. Always verify `EMBEDDING_VECTOR_DIMENSIONS` matches the actual model output before any data is stored.

### Silent failures compound
All four issues had been present since initial deployment. None surfaced until an outage forced actual fallback paths. Proactive monitoring (SDDC-60) is the corrective action — not hoping the happy path holds.

## Steps to Recreate This Stack from Scratch

1. **Clone and configure**
   ```bash
   git clone <honcho-repo> /opt/honcho
   cd /opt/honcho
   cp .env.example .env
   ```

2. **Set required `.env` values**
   - `DATABASE_URL` — Postgres connection string
   - `LLM_OPENAI_API_KEY` — set to a dummy value (e.g. `sk-dummy`) unless using real OpenAI; Honcho requires this env var to exist
   - `LLM_GROQ_API_KEY` — real Groq API key for fallback
   - All `MODEL_CONFIG__*` blocks for primary (Ollama) and fallback (Groq) per module
   - All `DIALECTIC_LEVELS__*` blocks for all five valid levels
   - `EMBEDDING_MODEL_CONFIG__*` pointing at `nomic-embed-text` via Ollama
   - `EMBEDDING_VECTOR_DIMENSIONS=768`
   - `LLM_REQUEST_TIMEOUT=45`
   - All four `FALLBACK__OVERRIDES__API_KEY_ENV=LLM_GROQ_API_KEY` pointers

3. **Patch `registry.py` for configurable timeout**
   Add at module top of `src/llm/registry.py`:
   ```python
   import os
   _LLM_REQUEST_TIMEOUT = float(os.environ.get("LLM_REQUEST_TIMEOUT", "45"))
   ```
   Replace all `timeout=600.0` references with `_LLM_REQUEST_TIMEOUT`.

4. **Initialize the database schema**
   ```bash
   docker compose up -d database
   docker compose run --rm api python scripts/provision_db.py
   ```

5. **Verify pgvector column dimensions**
   ```sql
   SELECT column_name, udt_name
   FROM information_schema.columns
   WHERE table_name IN ('message_embeddings', 'documents')
   AND column_name = 'embedding';
   ```
   If dimensions show 1536, run:
   ```sql
   ALTER TABLE message_embeddings ALTER COLUMN embedding TYPE vector(768);
   ALTER TABLE documents ALTER COLUMN embedding TYPE vector(768);
   ```
   Recreate HNSW indexes afterward.

6. **Pull required Ollama models on momoney**
   ```bash
   ssh momoney "ollama pull qwen2.5:14b-instruct-q4_K_M && ollama pull nomic-embed-text"
   ```

7. **Build and start the stack**
   ```bash
   docker compose build --no-cache
   docker compose up -d
   ```

8. **Validate**
   ```bash
   curl -s http://localhost:8000/health  # expect {"status":"ok"}
   docker compose logs api | grep -i error  # expect empty
   docker compose exec api python -c "from src.config import settings; print(settings.dialectic.levels.keys())"
   ```

## Consequences

**Positive:**
- Embeddings now functional for the first time since initial deployment
- Groq fallback will actually fire on Ollama failure
- No more 10-minute silent hangs on LLM calls
- All reasoning levels (`minimal` through `max`) route correctly to local Ollama
- Stack behavior is now fully observable and testable

**Negative / Risks:**
- Groq free tier is capacity-constrained — sustained Ollama outage will exhaust daily quota quickly
- `registry.py` patch is a local source modification that must be re-applied after upstream Honcho updates (tracked SDDC-57)
- VRAM contention on momoney remains a known risk; no automatic gaming-session detection in place yet (SDDC-60)
