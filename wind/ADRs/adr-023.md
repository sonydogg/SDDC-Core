# ADR-023: Claude as Async PM via n8n MCP — Direct Jira Integration Model

**Status:** Accepted (2026-04-05)

**Context:**

SDDC Core is a side project. The overhead of manual Jira ticket management (creating stories, updating status, logging progress) creates friction that slows down the actual technical work. The goal is to have Claude act as an async Project Manager — creating, updating, and triaging Jira tickets during development sessions and EOD standups — without requiring Trevor to touch Jira directly.

An initial attempt used an n8n AI Agent node (backed by Google Gemini) as an intermediary between Claude Code's MCP interface and the Jira API. This introduced unnecessary complexity: a second LLM was processing instructions that Claude had already reasoned about, adding latency, cost, and an unpredictable translation layer.

* **Constraint A:** Claude Code has MCP access to n8n's chat trigger endpoint. All Jira operations must be driven by Claude sending structured JSON — not by a second AI agent interpreting natural language.
* **Constraint B:** n8n's Jira node handles create, update, get, and comment operations cleanly. However, status transitions require the Jira Transitions API (`POST /rest/api/2/issue/{key}/transitions`), which the native n8n Jira node executes silently without error but does not actually apply when driven by expressions.
* **Constraint C:** The n8n HTTP Request node's `contentType: "json"` mode with `specifyBody: "string"` treats the body string as a form key rather than the request body. Raw JSON must be sent using `contentType: "raw"` with `rawContentType: "application/json"`.

**Decision:**

Replace the AI Agent + Gemini layer with a direct 5-node n8n workflow:

1. **Chat Trigger** — MCP-enabled entry point (n8n `chatTrigger` node)
2. **Parse Input** (Code node) — parses JSON from `chatInput`, strips null/empty fields, and resolves human-readable status names (`"in progress"`, `"done"`) to Jira transition IDs via a hardcoded map
3. **Route Operation** (Switch node) — routes on `operation` field: `create`, `update`, `get`, `transition`, `comment`
4. **Jira nodes** — one per operation: Create Issue, Update Issue, Get Issue, Add Comment (all `n8n-nodes-base.jira`)
5. **Transition Issue** (HTTP Request node) — calls `POST /rest/api/2/issue/{key}/transitions` directly with `contentType: "raw"` and `rawContentType: "application/json"`

The transition ID map (baked into the Parse Input Code node):

| Status | Transition ID |
|--------|--------------|
| idea | 11 |
| to do | 21 |
| in progress | 31 |
| in review | 41 |
| done | 51 |

Claude sends plain-English status names. The workflow resolves the correct transition ID internally.

**Consequences:**

* **Positive:** Single LLM in the loop — Claude reasons and decides, n8n executes. No Gemini overhead, no prompt-to-prompt translation errors.
* **Positive:** All five Jira operations (create, update, get, transition, comment) validated and working end-to-end.
* **Positive:** Workflow is MCP-enabled and published — Claude can drive it directly without any manual Jira interaction from Trevor.
* **Positive:** Transition IDs are resolved by name in the workflow — Claude never needs to know or send raw IDs.
* **Negative:** Transition IDs (11, 21, 31, 41, 51) are hardcoded in the Parse Input node. If the Jira project workflow is reconfigured (statuses renamed, reordered, or new ones added), the map will silently break — transitions will fail with a 400 from Jira with no clear indication of why.
* **Negative:** The `contentType: "raw"` workaround on the HTTP Request node is a fragile configuration. A future n8n upgrade that normalizes HTTP body handling could cause the transition operation to silently stop working (as happened with the `specifyBody: "string"` mode during development).
* **Negative:** The n8n Jira transition node (`operation: "transition"`) appears to execute without error when driven by expressions but does not apply the transition. This is a silent failure — there is no error surfaced. The workaround (HTTP Request node) must be maintained unless this n8n bug is resolved upstream.
* **Negative:** The Switch node (v3) requires an `options` block with `caseSensitive`, `leftValue`, and `typeValidation` fields on each condition. This is not documented and was discovered through a runtime error. Future n8n upgrades may change this requirement.

**MCP Configuration Note:**

The n8n MCP endpoint must be added to Claude Code's MCP server configuration (`~/.claude/settings.json` under `mcpServers`). The workflow must have `availableInMCP: true` in its settings block AND be published (active) for production execution. During development, `executionMode: "manual"` can be used to test unpublished versions. The MCP chat trigger uses `chatInput` as the input field — this is the key the Parse Input node reads from `$input.first().json.chatInput`.

**Known Future Breaking Change Risks:**

1. Jira workflow reconfiguration → transition ID map breaks silently
2. n8n HTTP Request node behavior change → raw body delivery breaks
3. n8n Jira node bug fix → transition node may start working, making the HTTP workaround redundant (low risk, easy to clean up)
4. n8n Switch node v4+ → conditions schema may change again
