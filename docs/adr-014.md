# ADR-014: Perforce Server Initialization and Recovery

**Status:** Accepted (2026-03-16)

**Context:**
Recreating the `sddc-p4d` Docker container can lead to a desynchronization between the internal Perforce metadata and the external `P4PASSWD_FILE`. To ensure the "Librarian" remains accessible to VS Code and n8n, a standardized recovery procedure is required.
* **Constraint A:** The `admin` user must always match the password defined in `./secrets/p4_admin_pass`.
* **Constraint B:** The server must maintain "Super" protections for the `admin` user to manage the `//depot/images` path on the `water` stone.

**Decision:**
We will use `docker exec` as the primary method for administrative overrides when the Perforce service is re-provisioned:
1. **Password Sync:** The `p4 passwd admin` command will be piped the contents of the secret file to re-align credentials.
2. **Security Level:** The server should be set to `security=2` to require passwords for all users, preventing unauthorized access to the SDDC Core via the 1666 port.

**Consequences:**
* **Positive:** Guaranteed alignment between the host-side secrets and container-side authentication.
* **Positive:** Restores VS Code connectivity immediately after a container reset.
* **Negative:** Requires manual execution of CLI commands if the container's internal volume (`/data` on the `earth` stone) is wiped.

---

**Technical Implementation:**
Initialization command:
`docker exec sddc-p4d p4 configure set security=2`
