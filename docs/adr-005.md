# ADR-005: The "Fifth Element" Filesystem Abstraction (Stones)

**Status:** Accepted (2026-03-16)

**Context:**
The Adamstown Intel Mini cluster (SDDC Core) currently operates on a single physical NVMe/SSD volume. To prepare for future expansion and maintain organizational clarity, we need a storage convention that separates functional workloads (Databases, Media, System).
* **Constraint A:** Maintain theme-consistency with the "Fifth Element" (SDDC Core).
* **Constraint B:** Ensure n8n and Docker paths remain static even if physical hardware changes.
* **Constraint C:** Support automated monitoring via the `/mnt/stones/*` wildcard.

**Decision:**
We will implement a **Logical Directory Abstraction** under `/mnt/stones/`. While these currently reside on the same physical partition, they are treated as independent "Stones" in all application logic:

1. **The Logical Mapping:**
   * `/mnt/stones/earth`: Persistent Data (Postgres/Config files).
   * `/mnt/stones/wind`: ephemeral/Active Data (n8n/Temp/Cache).
   * `/mnt/stones/water`: Mass Storage (Backups/Media).

2. **Transition Strategy:**
   * When physical disks are added, they will be mounted directly to these existing paths. This ensures no changes are required in n8n Workflows or Docker Compose files.

**Consequences:**
* **Positive:** Decouples "What" the data is from "Where" it physically sits.
* **Positive:** Monitoring logic remains consistent whether using one disk or ten.
* **Negative:** Since all stones share one physical disk, a "Disk Full" event on 'Water' will impact 'Earth' and 'Wind'. 
* **Negative:** Performance (IOPS) is shared across all stones until physical separation occurs.

---

**Technical Implementation:**
Current structure is managed via standard directory creation:
`mkdir -p /mnt/stones/{earth,wind,water}`
Permissions are set to allow the `n8n` and `postgres` containers access to their respective stones.
