# ADR-002: Persistent Storage via Elemental Stones

* **Status:** Approved
* **Date:** 2026-03-15
* **Deciders:** Trevor (Architect)

---

## 1. Context and Problem Statement
Local storage on the Intel Mini must be organized to balance high-speed database performance, frequent configuration changes, and large asset storage without creating a "monolithic" backup nightmare.

## 2. Decision Outcome
We are implementing a three-tier logical storage hierarchy mapped to physical drives, categorized by the "Elemental Stones" framework.

### The Hierarchy:
1. **Earth (IOPS Intensive):** Optimized for Databases.
   * *Target:* Postgres, Perforce metadata.
2. **Wind (High Frequency):** Optimized for Configuration.
   * *Target:* Docker Compose files, n8n binary data, Nginx Proxy settings.
3. **Water (High Volume):** Optimized for Large Assets.
   * *Target:* Perforce Depots, high-volume media, long-term logs.

## 3. Implementation (Docker Mapping)
All containers will mount volumes using consistent pathing:
* `/mnt/earth/data/[service]`
* `/mnt/wind/config/[service]`
* `/mnt/water/assets/[service]`

## 4. Consequences
* **Good:** Granular backup policies (e.g., backup Earth every hour, Water once a day).
* **Good:** Portability; "Stones" can be re-mounted to a new host with zero data loss.
* **Bad:** Requires discipline in initial Docker Compose volume mapping..
