# ADR-011: Unified Development Environment via VS Code

**Status:** Accepted (2026-03-16)

**Context:**
Managing a hybrid version control system (Git and Perforce) across multiple "Stones" requires a centralized, extensible IDE. The architect requires a tool that supports Markdown (for ADRs), YAML (for Docker), and HCL (for Packer) while maintaining active connections to both GitHub and the local Perforce Librarian.
* **Constraint A:** Prefer VS Code as the primary interface for all SDDC Core development.
* **Constraint B:** Support simultaneous Git and Perforce operations within the same workspace.
* **Constraint C:** Utilize Remote-SSH to edit files directly on the Intel Mini "Stones" filesystem.

**Decision:**
We will standardize on **VS Code** as the SDDC Core command center:
1. **Remote Operations:** All development is performed via the "Remote - SSH" extension, targeting the Intel Mini.
2. **Hybrid VC Integration:** - The **Perforce for VS Code** extension will manage heavy binary build logic and Image Factory assets.
   - The native **Git** integration will manage the documentation pipeline and public configuration.
3. **Environment Configuration:** A `.p4config` file will be placed in the root of the project directory to automatically switch Perforce settings based on the folder path.

**Consequences:**
* **Positive:** Reduces "Context Switching" between different terminals and apps.
* **Positive:** Provides immediate visual feedback on file status (e.g., Perforce "Checked Out" vs. Git "Modified").
* **Negative:** Requires careful management of `.gitignore` and Perforce "Ignore" files to ensure the two systems don't try to track each other's metadata.
* **Negative:** SSH overhead may impact performance when editing extremely large files directly on the Stones.

---

**Technical Implementation:**
Standard VS Code Workspace (`.code-workspace`) will be defined to include both the Git-tracked ADR directories and the Perforce-tracked Image Factory directories.
