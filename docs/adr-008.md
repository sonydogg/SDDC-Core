# ADR-008: Hybrid Version Control Strategy (Perforce & GitHub)

**Status:** Proposed (2026-03-16)

**Context:**
The SDDC Core utilizes both Perforce (P4D) and GitHub for version control. We need a clear boundary to define which system handles which workload to avoid data duplication and synchronization conflicts.
* **Constraint A:** Perforce is the primary internal repository for large binaries and local lab development.
* **Constraint B:** GitHub is the primary trigger for the n8n-to-Confluence documentation pipeline.
* **Constraint C:** Maintain the "Librarian" role for Perforce while leveraging GitHub's ecosystem.

**Decision:**
We will implement a **Two-Tiered Version Control System**:

1. **Internal Development (Perforce):**
   * All raw configuration files, local scripts, and large binary assets (ISO images, VM templates) are checked into the `mini-me-p4d` server.
   * Workspaces are mapped to the `/mnt/stones/` directories for direct filesystem interaction.

2. **Public Presentation & Automation (GitHub):**
   * Finalized Architectural Decision Records (ADRs) and "Stable" versions of the `docker-compose.yaml` are pushed to GitHub.
   * GitHub acts as the Webhook source for n8n to trigger the "Self-Healing" documentation sync (ADR-007).

**Consequences:**
* **Positive:** Combines the performance of local Perforce with the automation power of GitHub.
* **Positive:** Reduces GitHub storage bloat by keeping large binaries in the local Perforce depot.
* **Negative:** Requires a manual or automated "Bridge" to move files from Perforce to GitHub for publication.
* **Negative:** Potential for version mismatch if a file is updated in GitHub but not reflected in the local Perforce depot.

---

**Technical Implementation:**
Perforce Depots are stored on `/mnt/stones/water/p4depots`. GitHub remains the "Remote" for the `sddc-core` Git repository initialized on the Intel Mini.
